What is CRDT?

The Problem: Concurrent Edits

When multiple users edit the same document simultaneously, conflicts arise. Traditional solutions have major drawbacks:

Approach How it Works Problem
Locking Only one user can edit at a time Terrible UX, defeats collaboration
Last Write Wins Most recent change overwrites Data loss!
OT (Google Docs) Transform operations based on context Complex O(n²), requires central server
CRDT (Y.js) Mathematical properties guarantee merge ✅ No conflicts, works offline

CRDT Mathematical Properties

CRDTs guarantee eventual consistency through three mathematical properties:

A ⊕ B = B ⊕ A
Commutativity: Order of operations doesn't matter
(A ⊕ B) ⊕ C = A ⊕ (B ⊕ C)
Associativity: Grouping doesn't matter
A ⊕ A = A
Idempotency: Duplicate operations have no effect
Result: All replicas are guaranteed to converge to the same state, regardless of network conditions, timing, or order of operations. This is called Strong Eventual Consistency.

Y.js Deep Dive

Y.js assigns a unique ID to every character, making merges automatic and deterministic.

How Unique IDs Work

Each character gets: (clientID, clock)

User A (clientID: 1) types "Hi":
   'H' → { id: (1, 0), left: null,  right: (1,1) }
   'i' → { id: (1, 1), left: (1,0), right: null  }

User B (clientID: 2) types "!" after 'i':
   '!' → { id: (2, 0), left: (1,1), right: null  }

MERGE: Characters sorted by their unique IDs
   Result: "Hi!" ✅

No conflict because IDs are globally unique!

Why Sticky Sessions? WebSocket connections are stateful. Y.js document state lives in server memory. Redis pub/sub syncs updates across all instances, but routing the same user to the same server reduces latency.

Interview Q&A

Why CRDT over Operational Transformation?
CRDTs provide guaranteed eventual consistency without requiring a central server. Unlike OT (Google Docs), CRDTs work peer-to-peer and handle offline editing gracefully. Y.js uses unique IDs per character, making merges automatic. Trade-off is slightly higher memory, but simplicity and reliability are worth it.
How does real-time sync work?
When a user types, Y.js creates a CRDT operation with a unique ID (clientID + logical clock). This is encoded as a binary update (~10 bytes) and sent via WebSocket. The server broadcasts to all clients, who merge using CRDT rules. State vectors track what each client has seen, enabling efficient delta sync.
What if two users edit the same word?
Y.js handles this automatically. Each character has a unique ID, so both characters are preserved in deterministic order (client ID as tiebreaker). No data loss - worst case is merged text like "hweolrlod" which users can clean up. This is called intention preservation.
How would you scale to millions of users?
Current architecture supports horizontal scaling via sticky sessions and Redis pub/sub. For massive scale: document sharding by ID, read replicas for PostgreSQL, CDN for static assets, and connection pooling with PgBouncer.
